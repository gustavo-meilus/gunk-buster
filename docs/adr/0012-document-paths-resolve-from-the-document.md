# Document paths resolve from the containing document

Unanchored document path references resolve only from the containing document, matching Markdown semantics; a leading `/` explicitly anchors a path at the repository root. Relative mentions may use `/` or `\` separators, which normalize to `/`; drive-qualified and UNC paths are machine-local rather than repository claims and are skipped. A clean token must also carry a repository-path cue: an explicit anchor (`/`, `./`, `../` or Windows equivalents), a filename-like terminal extension, or an ancestor directory — other than the token's own resolution base — live in the current repository inventory. The resolution base is trivially live (the source document is indexed inside it), so excluding it prevents every unanchored, extension-less token from inheriting a cue from the document's own directory alone. This prevents MIME types, scoped packages, and other clean slash syntaxes from becoming claims. Tracked directories are valid targets when the current index contains at least one descendant; an explicit directory reference with no tracked descendants is dead, including in Markdown links. Gunk Buster does not try both bases or fall back to the repository root, because that would let ambiguous references appear live and preserve false rescues. Scan and Radar share one document-path extraction and resolution model that produces the normalized token, source location, anchor mode, and resolved target: Scan turns resolved candidate targets into reference assertions, while Radar judges unresolved mentions as dead-path claims. Markdown links reuse the same resolution primitive while retaining link-specific parsing. This intentionally changes behavior for root-intended paths in nested documents, which must add a leading slash and be called out in release migration notes.
